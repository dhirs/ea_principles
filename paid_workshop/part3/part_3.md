# Part 3 — Implementing the Catalogue (Enforcement Maturity)

## Where this sits

Part 2 was about how to **build** the catalogue — sourcing obligations and converting them into concrete, gated standards. Part 3 is about how to **enforce** it once built. A standard with no enforcement is a wish with a signature. This part is the operating model that makes standards stick — and because no org can start at full rigour, enforcement is **graded by maturity**.

## The core principle

The end goal of any gate is simple:

> **An EA gate consumes machine-generated evidence or enforces a non-bypassable control. It never consumes self-attestation.**

But real orgs can't begin there. So we grade enforcement strength along three rungs — **attest → verify → provide** — and apply the right rung **per control**, not org-wide.

| Level | Strength | What it is | Who produces the evidence |
|---|---|---|---|
| **1** | **Attest** | Project self-certifies coverage in a YAML | the project (declaration) |
| **2** | **Verify** | EA runs an adversarial test harness against the system | EA (measurement) |
| **3** | **Provide** | EA owns the control as non-bypassable platform infra | the platform (provision) |

## Level 1 — Self-certified (attest)

The project authors a **YAML per guardrail**, declaring how it has handled each threat in the agreed taxonomy (OWASP-LLM / ATLAS):

```yaml
guardrail: refund-triage-agent
owner: jane.doe@firm.com          # named, accountable sign-off
standard: ST-GENSEC04-12
threats:
  LLM01_prompt_injection: { disposition: mitigated, evidence: "system/user role separation + input sanitiser" }
  LLM02_insecure_output:  { disposition: mitigated, evidence: "output schema allow-list" }
  LLM06_excessive_agency: { disposition: mitigated, evidence: "refund tool gated at £X, else route-to-human" }
  LLM04_data_poisoning:   { disposition: not_applicable, justification: "no fine-tuning; vendor FM" }
  # ... every category in the taxonomy must carry a disposition
```

**What CI enforces here:** not efficacy — **completeness**. The pipeline rejects the YAML unless **every** taxonomy category carries a disposition (`mitigated` / `accepted` / `not_applicable` + justification). That turns L1 from an honour system into **enforced coverage of the conversation**.

**Why it still has teeth despite being unauditable:** the named sign-off is a durable record. Wired to **post-incident review**, it becomes accountability — *"you certified you handled injection; you didn't."* Without that consequence, L1 is theatre. The deterrent is the accountability, not the YAML.

**Honest limit:** L1 proves a project *thought about* every threat and *claimed* a disposition. It cannot prove the claim is true.

## Level 2 — Independently verified (verify)

EA owns an **adversarial eval harness** and runs it *itself* against the deployed system. For prompt injection: fire the OWASP/ATLAS injection corpus at the endpoint; the gate is *"0 successful instruction overrides."*

This is **behavioural proof, machine-generated by EA** — the project can't fake it, because EA attacks their actual running system, not their checklist.

**Coverage limit:** only **probe-testable** controls qualify — injection, jailbreak, output toxicity, hallucination/grounding. **Structural and process controls** (logging completeness, provenance, human-oversight process) are not probe-testable and stay at L1 attestation or move to L3 provision.

## Level 3 — Centrally provided (provide)

EA stops verifying the project's implementation and instead **owns the control as shared infrastructure** — a centralized **SDK / AI gateway** that performs injection filtering, output scanning, tool-call authorisation, and logging. Compliance is no longer *"did you build it?"* but *"are you calling through the sanctioned client?"*

**Critical caveat — the SDK alone is NOT Level 3.** A published SDK is opt-in: a library a team can decline to import. If a project can still call the model provider directly, L3 silently collapses back to L1 (you're trusting they used it). The real control is the **egress / IAM policy that blocks every path to the model except through the sanctioned client.** L3 = SDK **+ platform-enforced routing**, or it isn't L3.

**Coverage limit:** the gateway owns **horizontal** controls (filtering, logging, tool-auth). **Domain-specific** controls — *"refund ≤ £X,"* business eligibility logic — are application-specific and stay with the project. L3 is not 100% coverage.

## Cross-cutting mechanism 1 — CI (deploy-time, every level)

CI is **not a level** — it is the rail present at *all* levels. What it enforces **escalates** with the level:

- **L1** → YAML exists and every taxonomy category is dispositioned.
- **L2** → the eval harness passes (0 overrides).
- **L3** → routing proof (traffic goes through the gateway) + a **signed attestation**; pair with build provenance (SLSA-style) so the deployed artifact is provably the one that passed.

No green, no deploy — at every level.

## Cross-cutting mechanism 2 — Platform policy (runtime partner)

CI is a **deploy-time** gate; a team can pass CI then reconfigure. L3's non-bypassability lives at **runtime** — egress policy and telemetry. So the enforcement plane is always **CI (deploy) + platform policy (runtime)**, paired. Runtime conformance monitoring also confirms the control stays *on* in production (every inference logged, every tool call authorised, detection firing) — compliance as a live signal, not a one-time gate.

## Five design rules (so the maturity model survives reality)

1. **Levels coexist; they don't replace.** This is a capability set, not a staircase you climb and leave behind. Pick the rung **per control**. Some controls — training-corpus provenance, copyright, human-oversight process — are **never probe-testable or centralizable** and stay at L1 attestation *permanently*, even at full maturity.
2. **An SDK is not L3 without egress enforcement.** Publish a library and you have L1 with extra steps. The control is the routing policy.
3. **L1 must machine-check completeness and wire to accountability.** Enforce that every threat is dispositioned, and tie the sign-off to post-incident review — or it's theatre.
4. **Maturity is a per-concern portfolio, not an org-wide scalar.** You will be L3 on prompt-security and L1 on fairness because tooling matures at different rates. Track the level **per control** and report the *distribution* — "we're a Level 2 org" hides the truth.
5. **CI (deploy) + platform policy (runtime), always paired.** Neither alone holds the line.

## Worked example — prompt injection across the levels

| Level | What the project does | What EA checks (the gate) |
|---|---|---|
| **L1** | Declares `LLM01: mitigated` in the guardrail YAML, named owner | CI: every taxonomy category dispositioned; sign-off present |
| **L2** | Builds whatever defence it likes | EA fires the OWASP/ATLAS injection corpus at the endpoint → 0 overrides to pass |
| **L3** | Routes all model calls through the EA gateway | Egress policy blocks direct calls; gate = routing proof + signed attestation; gateway filters injection centrally |

## One-line summary

**Three enforcement strengths — attest, verify, provide — applied per control (not org-wide), all riding CI at deploy and platform policy at runtime, with self-cert wired to accountability and the SDK wired to egress enforcement, or neither earns its level.**
