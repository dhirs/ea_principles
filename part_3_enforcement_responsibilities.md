# Part 3 (companion) — Who Does What at Each Enforcement Level

Companion to `part_3.md`. That doc defines the three enforcement strengths — **attest → verify → provide**.
This one assigns **ownership**: for each level, who does what across **EA** (which splits into Lead AI
Architect + Governance/ARB), **Engineering**, **Product**, and the **Project team**. Built from the
workshop discussion on running enforcement when no platform exists yet.

## The four functions (recap)

- **EA — Lead AI Architect** — owns *what a standard says* and *what "pass" means* (the criteria,
  thresholds, which controls are probe-testable). Defines; does not build or run.
- **EA — Governance / ARB** — owns the *verdict and the gate*. Consumes evidence, certifies pass/fail,
  blocks release, holds waiver authority. Reports out to the CRO (independent of shipping).
- **Engineering** — *builds* the platform tooling (SDK, harness, corpus framework, scorer). Builds; does
  not run it against projects or judge results.
- **Product** — *operates* the capability as a service with an SLA: scheduling, turnaround, adoption,
  enablement, cost pool, reporting. Runs the machine; does not own the verdict.
- **Project team** — provides the workload-specific surface (endpoint, contracts, invariants, adapter)
  and remediates failures.

> The governing rule across all levels: **build / operate / define / judge are four different owners, on
> purpose.** If the group that runs a check also rules on its result, the independence that makes the
> level worth anything is gone.

---

## Level 1 — Attest (self-certification)

**What it is.** The project self-certifies, at the ARB, that it has handled every threat in the agreed
taxonomy (OWASP-LLM). A guardrail YAML per workload, every threat carrying a disposition
(`mitigated` + evidence / `accepted` / `not_applicable` + justification), with a named, accountable owner.
No SDK, no harness needed — enforceable today.

**The process (the architect drives applicability; the ARB attacks the soft answers):**
1. Scope the workload — pattern (LLM/RAG/agent), serving paradigm, risk tier. The catalogue's
   `applicability` / `serving_paradigm` / `maturity_level` fields filter the standards automatically — it
   is metadata-driven, not the architect's opinion.
2. Disposition every standard that survives the filter.
3. Named owner signs.
4. ARB reviews: completeness is mechanical (no blanks); the value-add is challenging the
   `not_applicable`s and `accepted`s — that is where projects hide.

**What CI enforces (if any pipeline exists):** completeness only — the YAML exists and every category is
dispositioned. Not efficacy. If no CI exists, the ARB *is* the gate (manual check).

| Function | Responsibility at L1 |
|---|---|
| **EA — Architect** | Owns the threat taxonomy + the standards catalogue; defines what a valid disposition looks like; sets the applicability metadata that filters the list. |
| **EA — Governance/ARB** | Runs the ARB review; checks completeness; **challenges the N/A and accepted dispositions**; certifies / conditions / rejects; wires the signed attestation to post-incident review. |
| **Engineering** | Minimal. Optionally a ~20-line completeness linter for the YAML. No platform required. |
| **Product** | Owns the attestation template + the submission process; tracks which workloads have a signed attestation. |
| **Project team** | Authors the guardrail YAML; dispositions every relevant standard; names the accountable owner. |

**Honest limit:** L1 proves the project *thought about* and *claimed* a control — never that the claim is
true. Teeth come from the signature + post-incident accountability, not the YAML.

## Level 2 — Verify (independent adversarial testing)

**What it is.** EA tests the project's **running system** itself. An adversarial harness fires an attack
corpus at the deployed endpoint and scores the responses against declared invariants. Behavioural proof,
machine-generated by EA — the project can't fake it because the attack hits their real system, not their
checklist.

**Worked shape (returns-triage agent, prompt injection on the entry prompt):**
- EA fires real-shaped requests with the attack in the one untrusted field (`return_reason`); structured
  fields held constant.
- Each corpus case is `payload + the field to inject + the oracle` (e.g. `must_not: AUTO_APPROVE`).
- The corpus is **template-generated, not hand-written** — ~25 attack templates × envelopes × obfuscation
  variants + public corpora (OWASP/ATLAS, Garak/PyRIT) → hundreds of cases. Labels inherit per template,
  not per case.
- Pass = 0 successful overrides across the attack set **and** false-positive rate on a benign control set
  under threshold.

**Why not "the project just pentests itself":** rewarded-on-shipping means it gets cut; self-testing is
grading your own homework; 500 projects shouldn't each invent (and let rot) their own attack set; prompt
injection / jailbreak / grounding is a different threat class than classic pentest; and "passed the
central corpus, 0 overrides, run by EA" is one auditable claim where "we pentested it" is 500 different
ones. EA standardises **the corpus, the bar, and the independence** — not *that* you test.

**The infra built once, centrally (the "repeatability cost" justification):**
1. **Attack corpus library** — curated, versioned, kept current; new attack added once → all workloads
   re-tested.
2. **Harness runner** — fires the corpus at an endpoint, parallelised, rate-limited, collects responses.
3. **Scoring / oracle engine** — judges pass/fail; for fuzzy checks (grounding, toxicity) an LLM/NLI judge
   with owned thresholds.
4. **Test environment + cost pool** — hundreds of metered inference calls × workloads × re-runs; funded
   and metered centrally, not 500 separate bills.
5. **Results store + evidence dashboard** — per-workload verification record, history, audit-ready.
6. **Scheduling + gate wiring + secrets** — trigger on release, wire the verdict to the deploy gate,
   manage test credentials.

**The canonical-format problem (different projects, different return signatures):** the harness does **not**
force a uniform API. The project supplies a thin **adapter** — a `test_contract.yaml` declaring JSONPaths
from its native response to a canonical `{decision, response_text, tool_calls}`, plus its invariants
(`decision_must_not: [AUTO_APPROVE]`). A code adapter function is the fallback for awkward shapes. The
canonical format lives in the adapter, ~15 lines, written once.

**What stays with the project vs central:** project owns the **staging endpoint**, the **invariants/oracle**,
**test-data seeding**, and the **~15-line adapter** — irreducible, because only the project knows its
response shape and what its agent must never do. Everything else (corpus, runner, scorer, compute, record)
is central — the expensive 95%.

**Coverage limit:** only **probe-testable** controls reach L2 — injection, jailbreak, output toxicity,
grounding. Structural/process controls (logging completeness, provenance, human-oversight) are not
attackable from outside and stay at L1 or move to L3.

| Function | Responsibility at L2 |
|---|---|
| **EA — Architect** | Defines "pass": what counts as a successful override, the threshold (0), which controls are probe-testable, what the oracle must contain. Authors once; doesn't run. |
| **EA — Governance/ARB** | Owns the verdict and the gate. Consumes the harness report as machine evidence, certifies pass/fail, blocks release, holds waiver authority. Independent of shipping. |
| **Engineering** | **Builds** the harness, the corpus library + generator, the scoring/oracle engine; keeps the corpus current as new attacks emerge. Builds, doesn't run or judge. |
| **Product** | **Operates** it as a service: schedules runs against the project's staging endpoint, owns turnaround SLA, delivers the report, runs the cost pool, supports the project, tracks verification coverage. |
| **Project team** | Provides the staging endpoint + test creds, the input contract, the invariants/oracle, the ~15-line adapter; remediates failures and requests re-test. |

## Level 3 — Provide (centrally owned, non-bypassable)

**What it is.** EA stops verifying the project's implementation and **owns the control as shared
infrastructure** — a central SDK / AI gateway performing injection filtering, output scanning, tool-call
authorisation, logging. Compliance flips from *"did you build it?"* to *"are you calling through the
sanctioned client?"*

**Critical caveat — the SDK alone is NOT L3.** A published SDK is opt-in; if a project can still call the
model provider directly, L3 collapses to L1. **L3 = SDK + platform-enforced egress/IAM routing** that
blocks every path to the model except through the sanctioned client — or it isn't L3.

**Coverage limit:** the gateway owns **horizontal** controls (filtering, logging, tool-auth). **Domain-
specific** controls — "refund ≤ £X", business eligibility — stay with the project. L3 is not 100% coverage.

| Function | Responsibility at L3 |
|---|---|
| **EA — Architect** | Owns *what the SDK/gateway must enforce* — the standard content baked into the central client and the routing policy's required behaviour. |
| **EA — Governance/ARB** | Gate = routing proof (traffic through the gateway) + signed attestation + build provenance. Certifies; owns waivers for genuine gaps. |
| **Engineering** | **Builds and maintains** the SDK, the gateway, the egress/IAM enforcement, filtering/scanning/tool-auth, the migration tooling (shims, codemods). |
| **Product** | **Operates** the gateway as a product: SLA, run/on-call, versioning/deprecation, **adoption & brownfield migration**, FinOps/chargeback, the readiness gate, day-to-day waiver intake. Holds the **enable-before-enforce** sequencing. |
| **Project team** | Routes all model calls through the sanctioned client; keeps domain-specific controls; migrates legacy call sites onto the SDK. |

## Cross-cutting (all levels)

- **Levels coexist; pick the rung per control.** A workload is L3 on prompt-security, L2 on grounding, L1
  on fairness — simultaneously. Report the *distribution*, not an org-wide scalar.
- **CI (deploy) + platform policy (runtime), always paired.** CI escalates with the level (L1 completeness
  → L2 eval passes → L3 routing proof). Runtime egress/telemetry holds non-bypassability. Neither alone
  holds the line.
- **Enable before enforce (Product owns the sequencing).** A gate flips to *required* only once the
  capability is ready and a migration path exists. New projects start on the SDK because it's the
  scaffold/starter template — the compliant path is the path of least resistance. Greenfield-by-default;
  brownfield is a funded retrofit workstream.
- **Who flips the switch:** Product proposes readiness; **Governance** makes the gate required and holds
  waiver authority. Architecture says what passing means; Engineering builds it; Product runs it.

## One-line summary

**Three strengths — attest, verify, provide — each with the same ownership spine: Architecture defines the
bar, Engineering builds the machine, Product runs it as a service, Governance owns the verdict, and the
project owns only the thin workload-specific surface (dispositions at L1, endpoint + oracle + adapter at
L2, routing through the sanctioned client at L3).**
